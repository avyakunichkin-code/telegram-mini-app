# Система уровней персонажа и опыта (Game Mode)

Живой **дизайн-документ**: формулы, источники XP, дорожная карта разблокировки механик и связь с **`event_tier`**. Изменять баланс **здесь и в матрице** [`catalogs/XP_EVENTS_ACTIONS_MATRIX.md`](catalogs/XP_EVENTS_ACTIONS_MATRIX.md), затем синхронизировать код (**константы / сиды**) и смежные спеки.

**Инвариант:** `GameProfile.level` и `GameProfile.xp` описывают **ролевой рост игрока**; строка **`gamification_level`** и **`score`** в `FinanceOverview` — **отдельная** финансовая условная градация, не управляет выпадением событий ([`SPEC_mvp-11-progression-events` §3](../features/SPEC_mvp-11-progression-events.md)).

---

## 1. Зачем и что решаем (How Might We)

Когда события привязаны к **развитию игрока**, игрок должен понимать: **за что даётся опыт**, **когда он чувствует рост**, **что теоретически открывается дальше** — без смешения с «рейтингом финграмотности» из overview.

---

## 2. Идеи (расширение) → три режима разблокировок

Чтобы спроектировать механику, полезно отделить три уровня влияния **уровня персонажа**:

| Вариант | Суть | Плюсы | Минусы |
|---------|------|-------|--------|
| **A. Только события** | Только фильтрация **`event_tier`** + контент событий; механики (инвестиции, покупки) всегда доступны по экономике/шаблону | Быстро в коде MVP 1.1 | Нет «ощущения уровней» помимо текстов событий |
| **B. Жёсткие гейты на API** | Каждый эндпоинт дорогих механик проверяет `profile.level ≥ unlock_level`; UI прячет/блокирует | Сильнейший игровой смысл «открылся депозит» | Больше дифф по роутерам и UX; нужна таблица разблокировок |
| **C. Гибрид «мягкий + жёсткий»** | Сначала **подсказки** и ограничения UI; затем включаются **404/400 от сервера** для читерства | Поэтапный спуск риска | Два слоя поведения, нужна синхронизация |

### Рекомендация (**утверждено продукции 2026-05-17**): **вариант C (гибрид)**

Последовательно: **видимость + подсказки в UI**, затем **серверные гейты** дорогих механик (см. [`PLAN_level-xp-progression`](../../plans/PLAN_level-xp-progression.md) фазы 3–4). Вариант **A** дастся только как временный шаг между релизами при дефиците времени разработки.

## 3. Целевая карта разблокировок (концепт → данные в код)

Ориентация по прогрессии из [evolution §II](../../vision/ideas/money-quest-evolution-after-mvp.md): плавное открытие механик. Ниже **целевая** привязка уровней (число уровней ориентировочное **до 12** первой версии экономики игры):

| Целевой `character_level` (разблок с) | Механика / смысл | Примечание |
|----------------------------------------|-----------------|------------|
| 1 | Базовые действия месяца, зарплата, простые траты-события | уже в MVP до G1 |
| 2 | Подушка (`safety_fund`) активно через UX как «первая защита» (если ещё не выделено) | визуал/тон, не блок API при необходимости |
| 3 | Простые **инвестиции**: депозит | API-gate см. PLAN фаза 3 |
| 4 | Облигации или второй класс активов инвестирования | балансируется |
| 5 | Страхование (вид полиса можно дробить) | см. связь событие↔страховка backlog |
| 6+ | Шаблоны активов высокого уровня, «тяжёлые» события `event_tier ≥ 5`, комбинации долг+доход | контент-dependent |

Эта таблица **не** должна противоречить шаблону старта: если blueprint уже выдал ипотеку на первом месяце, **блокировать обслуживание долга через level нельзя** — ограничивать только **новые действия** (новый кредит, новый продукт, новый актив из каталога), либо вводить **исключение** «уже есть в профиле → разрешено». Это **§8 инвариант**.

## 4. Формула порогов XP (уровневая прогрессия)

### 4.1. Формула MVP (закрепить в коде как v1)

Согласовать с действующими `game_period.py` / `period_actions.py` после рефакторинга (**одна** функция, см. `SPEC_mvp-11` §5):

\[
\text{need}(L) = 100 + \max(0,\, L-1) \times 50 \quad\text{для перехода } L\rightarrow L+1
\]

Накопленный `xp` профиля в пределах уровня: целое **≥0**; поддержка многократного level-up за одно действие сохраняется.

### 4.2. Альтернативные кривые (idea-refine, не блокируют v1)

| Вариант | Идея | Когда брать |
|---------|------|-------------|
| **Полином / степенная** \(\text{need} \sim L^k\) | более крутая середина игры | если «уровни улетают» слишком быстро после волны XP из событий |
| **По сумме периодов** | синхрон с `period_index` | если хотим «месяцы и уровень» синхронны визуально |
| **По времени календаря** | вне MVP TMA-сессий | не рекомендуется сейчас |

### 4.3. Ориентир темпа (для смоуков)

Целевой диапазон **утверждён продукцией 2026-05-17:** при «активном» игроке (зарплата большинство периодов, участие в событиях) ориентиться на **\(12\text{–}18\)** игровых **периодов** до **character level 5**. Отклонение — сигнал к пересмотру конфигурации XP в матрице или изменению сумм `effects_json`.

*(Ранее обсуждённый порог 8–15 остаётся запасным ориентиром для смоуков «быстрейшего прохождения».)*

## 5. Система начисления XP: принципы

1. **Прозрачность:** любое автоматическое начисление фиксируется в понятности для игрока — **в MVP 1.1 ограничиваемся кратким тостом** при level-up / крупных начислениях; расшифровка постфактум («лента источников») — отдельная фича после 1.1.
2. **Не наказывать тихими нулями без причины:** если действие дорогое но полезное для учёба (страховка), XP может быть маленьким, но предсказуемым из матрицы.
3. **События — основной вертикальный рычаг вариативности:** суммы задаются в данных (`effects_json.xp_delta` + пересмотр сидов), не хардкодом в каждом выборе.
4. **Отрицательный XP запрещён:** штрафы за ошибки только **денежные / Lifestyle / статусы экономики**, не через `xp_delta < 0`.
5. **Анти-фарм:** капсы на частые действия можно ввести в **фазе баланса** (напр. вклад пополнений в период > N раз — без доп. XP) — см. PLAN фаза 4.
Подробная **пер-источниковая таблица** поддерживается в [**`catalogs/XP_EVENTS_ACTIONS_MATRIX.md`**](catalogs/XP_EVENTS_ACTIONS_MATRIX.md) и синхронизируется после каждого балансового изменения.

### 5.1. Черновые значения источников (наследие кода MVP)

На момент 2026-05 наблюдались в коде (проверять при каждой правке):

| Источник (черновик ключа матрицы) | Порядок величины XP (до обновления матрицы) |
|-----------------------------------|--------------------------------------------|
| claim-salary (примерно) | +20 бонус в одном месте смешан с переходом периода (`period_actions`) |
| вклад подушку / частичные действия периода | +5/+10 паттерны |
| завершение периода базово | ~5–30+ смоук-бонусы зависящие от сбережений снимка |

**Норматив:** после создания конфиг-перечисления в коде строки кодов должны совпасть со столбцом **XP** матрицы (поколение ошибок исключается через unit-тест на константы).

## 6. Связь с **`event_tier`**

| Сущность | Роль |
|----------|------|
| `character_level` (профиль) | Ограничивает допустимый пул через окно выпадания + будущее открытие механик |
| `event_tier` (определение) | Отражает сложность/глубину **конкретной карточки** событий |
| `xp_delta` на выборе | Альтернативный усилитель последствий; не должен доминировать над матрицей «базового» месяца без балансового прогона |

## 8. Инварианты против «сломали шаблон»

Если объект экономики уже существует в профиле (долг, актив, инвест-позиция из blueprint), сервер:

- **не** запрещает обслуживание/конец периода из-за низкого уровня;
- может запрещать **новые действия этого класса** до разблока (ПК **B или C** §2).
