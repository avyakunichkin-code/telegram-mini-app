# DL1 — ревью согласованности математики

Ревью как **финансовая модель** (не код). Итог: эпик реализуем, но без §4.0 «Канон полей» и правок ниже в реализации будут расхождения UI ↔ ledger ↔ победа.

---

## 1. Согласовано (можно не менять)

| Тема | Вердикт |
|------|---------|
| Формула аннуитета §4.1 | Стандартная \(A = P \cdot \frac{r(1+r)^n}{(1+r)^n-1}\), \(r=0 \Rightarrow P/n\) — **корректно** |
| Закрытие хвоста в последнем периоде | `principal_part = min(payment - interest, P)` — **нужно явно в коде** |
| Путь A: бюджетная тождественность | `down_payment + original_principal = purchase_price` при `asset_value = purchase_price` — **OK** |
| Продажа: `cash_net = max(0, sale - payoff) - top_up` | **OK** при однозначном `payoff` |
| Закрытие долга вручную (`delete_liability`) | `payoff = overdue + total_debt` — **совпадает** с продажей |
| Страховка: `expires = started + term_periods` | 12 периодов при премии пока `period_index < expires` — **согласовано** с `insurance/service.py` |
| Legacy interest-only | Параллельный режим без амортизации — **не конфликтует**, если `payment_mode` разводит логику |

## 2. Критические несогласованности (исправить до кода)

### 2.1. Что такое `total_debt` (блокер)

| Источник | Формулировка |
|----------|----------------|
| SPEC §4.1 | `P` — остаток **тела**; проценты в `interest_part`, не в `P` |
| ADR-010 §2 п.2 | «`total_debt` (тело **+ начисленные проценты**…)» |
| SPEC §5.3 | `payoff = overdue + total_debt` «и проценты, если учтены в total_debt» |

**Проблема:** при аннуитете начисленные, но **неуплаченные** проценты не должны одновременно входить в `overdue_amount` (как недоплата графика) **и** добавляться к `total_debt` — **двойной учёт**.

**Канон (рекомендация):**

- `total_debt` := **только остаток основного долга** \(P\).
- `overdue_amount` := **вся недоплата** по графику (в т.ч. просроченные проценты и тело в составе платежа).
- **Payoff при продаже/закрытии:** `payoff = overdue_amount + total_debt` (без отдельного «хвоста процентов»).

→ Правка ADR-010 §2 и SPEC §5.3 (см. коммит доков).

### 2.2. Частичная оплата в `period_end` vs аннуитет

Сейчас в коде: `unpaid = due_total - paid`, тело **не** уменьшается.

Для аннуитета нужно правило:

- **Снижение `total_debt`** только если `paid >= monthly_payment` (полный платёж периода) **или** явная аллокация (сначала проценты из платежа, потом тело).
- При `0 < paid < monthly_payment`: весь `unpaid` → `overdue_amount`, **P без изменений**.

Иначе DL1-AC-2 («тело уменьшается каждый закрытый период») ложно при частичных списаниях.

### 2.3. `periods_paid` (блокер для срока)

SPEC: «включая просроченные попытки» — **математически неверно**: при неоплате срок кредита «съедается», а тело не тает.

**Канон:** `periods_paid += 1` только при **полной** уплате `monthly_payment + carry_overdue` (или при закрытии долга).

Оставшиеся периоды: `n_rem = term_periods - periods_paid`.

### 2.4. Prepay vs просрочка

| | Продажа / полное закрытие | Prepay §4.3 |
|--|---------------------------|-------------|
| Порядок | overdue → тело | «не сбрасывает overdue» |

**Проблема:** досрочное погашение тела при живой просрочке — нетипично и путает игрока.

**Канон:** как у банка — **waterfall:** (1) `overdue_amount`, (2) остаток `amount` на уменьшение `total_debt`, (3) пересчёт `monthly_payment` на `n_rem`.

### 2.5. Фиксированный платёж vs пересчёт каждый период

§4.1 описывает шаг после платежа, но не фиксирует:

- `monthly_payment` задаётся **при выдаче** из \((P_0, r, n_0)\) и держится до prepay/реструктуризации;
- **не** пересчитывается каждый месяц от нового \(P\) (иначе это не классический аннуитет, платёж «плывёт»).

После prepay: новый \(P\), тот же `n_rem` → новый `monthly_payment`.

## 4. Числовой smoke-test (путь A)

Покупка: цена 10 000 000 ₽, взнос 2 000 000 ₽, кредит 8 000 000 ₽, 12% год., 240 мес.

- \(r = 0.01\), \(n = 240\), \(P = 8\,000\,000\)
- Платёж \(A \approx 88\,000\) ₽/мес (порядок величины)
- Через 120 мес. при полной оплате \(P \ll 8\,000\,000\)
- Продажа при `asset_value = 10\,000\,000`, `total_debt = 5\,000\,000`, `overdue = 0` → на cash **5 000 000** — OK
- Продажа при `asset_value = 4\,000\,000`, `total_debt = 5\,000\,000` → top-up **1 000 000** с cash — OK
