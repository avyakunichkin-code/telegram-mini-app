# Spec: вкладка «Аналитика» (Money Quest)

Документ фиксирует **вижн и дорожную карту** экрана аналитики: что показываем игроку, зачем (обучение + обратная связь), какие данные нужны от бэкенда.

Текущий MVP вкладки строится на полях `GET /api/finance/overview` и текстовых блоках ниже не хватает — пункты помечены **(будущее)**.

**Согласование с продуктом:** целевая модель победы и метрик — [evolution §II](../vision/ideas/money-quest-evolution-after-mvp.md); «победа недоступна до 7-го периода» — из `period_index` в overview.

---

## 1. Цели экрана

| Цель | Как поддерживается |
|------|--------------------|
| **Осмыслить месяц** | Свести доходы, обязательные выплаты, обслуживание в одну картину |
| **Увидеть риск раньше игрового провала** | Просрочки, долговая нагрузка, отрицательный поток |
| **Привязать прогресс к победе** | Подушка к цели, условия win |
| **Мотивация без морализаторства** | Графики и тренды, а не текст «ты плохо потратил» |

---

## 2. Принципы UX (совмещение визуала и логики)

1. **Один главный KPI на блок** — не перегружать карточки цифрами без подписей единиц.  
2. **Те же понятия, что в экономике месяца** — термины совпадают с главным экраном и вкладкой «Финансы».  
3. **Цвет как смысл**, согласованный с брендбуком: изумруд / фиолетовый акцент / danger для риска, без лишней «радуги».  
4. **Пустые и ранние партии**: короткая подсказка «данные соберутся после первых периодов» там, где нет истории.

---

## 3. Контент по фазам

### Фаза A — уже возможна (обзор + простые столбики)

**API:** `GET /api/finance/analytics/timeseries?limit=48` — по закрытым периодам сохранённые снимки (`period_economy_closings` при завершении месяца) + текущая **проекция** по живому профилю. В ответе также `clean_period_streak`.

- **Прогресс подушки** к цели победы (доля или «заполненная дорожка»).  
- **Сравнение потоков** за *текущую модель месяца*: доход vs платежи по долгам vs обслуживание активов (нормализация к общему масштабу, как во фронте).  
- **Метрики риска**: `liabilities_to_income_ratio`, `total_overdue_amount`, чистый поток.  
- **Уровень / XP / период** — как контекст «где ты в партии».

### Фаза B — нужны серийные данные **(будущее)**

- **График баланса и подушки по периодам** (`PeriodSnapshot`, агрегаты по месяцам).  
- **Водопад месяца**: зарплата → платежи → события → итог.  
- **Стек обязательств**: как меняют долг платежи (остатки по кредитам).  
- **Доходность инвестиций** нарастающим итогом.

### Фаза C — «умные» элементы **(будущее)**

- **Сценарий «если»: что будет, если следующий месяц без событий / без зарплаты** (лайт-симуляция на клиенте от текущего снимка).  
- **Подсказки-инсайты** (правила без ML): например «платежи > 45% дохода — зона давления».

### Фаза D — мета и сравнение **(опционально)**

- Сравнение с **«идеальной» точкой победы** по всем условиям.  
- **Личная статистика** за N партий (без акцента на лидерборды до готовности анти-abuse).

---

## 4. Типы визуализаций и зачем

| Вид | Что объясняет |
|-----|----------------|
| **Горизонтальные bar / bullet chart** | Соотношение статей месяца (компактно в TMA) |
| **Sparklines** по периодам | Инерция долга и рост подушки |
| **Gauge / шкала победы** | Единое поле цели MVP |
| **Таблица «последние 6 периодов»** с цветными дельтами | Без графика тоже читается |
| **Теплокарта дня/недели** **(маловероятно)** | Только если появится суточная зернистость |

---

## 5. Источники данных (бекенд / модели)

| Данные | Сейчас | Нужно для B+ |
|--------|--------|--------------|
| Текущий снимок | `FinanceOverview` | — |
| История месяцев | фрагментарно через снимки/транзакции | явный **`GET .../finance/timeseries`** или расширение overview |
| События | по выбору в периоде | агрегат «доходы/расходы от событий за период» |
| Инвестиции | позиции + начисления | история **`principal`/купон по периодам** |

---

## 6. Техника и платформа (Telegram Mini App)

- Избегать тяжёлых SVG-чартов там, где достаточно **CSS-баров** и нативного скролла секций.  
- Опционально библиотека **lightweight-chart** (графики) только если появится чёткая потребность — не путать с устаревшим игровым «режимом light».  
- **Доступность**: для баров добавлять текстовые дубли (процент + сумма) для скринридеров.
