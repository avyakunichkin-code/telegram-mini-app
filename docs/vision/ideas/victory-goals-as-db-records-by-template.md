# Victory goals как записи в БД (по template_key)

## Problem Statement
Как нам хранить и управлять целями (Victory v2 / tutorial chain) так, чтобы:
- минимизировать ошибки и рассинхрон (порядок целей, `requires_mechanics`, доступность механик),
- упростить сопровождение (меньше правок “монолитного JSON” и миграций-накладок),
- подготовить почву для будущей админки,
при этом сохранить понятную привязку целей к шаблону игры.

## Recommended Direction
Перейти на хранение целей в БД как **табличных записей, привязанных к `template_key`**, с гибридной схемой:
- “каркас” цели — в колонках (ключи/порядок/включённость/зависимости),
- типо-специфичные параметры цели — в `params JSONB` (строго валидируемом по `type` на уровне backend).

Это направление сохраняет гибкость (добавлять новые типы целей без миграций схемы под каждую мелочь), но даёт высокую консистентность и возможность автоматической проверки.

## Key Assumptions to Validate
- [ ] Хранение “каркаса” цели в колонках + `params JSONB` позволит поддерживать новые типы целей без деградации качества данных (через обязательную валидацию по `type`).
- [ ] Привязки только к шаблону (`template_key`) достаточно на ближайшее время (без overrides на профиль).
- [ ] Можно сохранить API-контракт (`overview.victory`), изменив только источник данных на backend.

## MVP Scope
### 1) Таблица целей (минимальный DDL)
Ввести таблицу (название условное) `victory_goals`:
- `id` (pk)
- `template_key` (fk/строка, ключ шаблона)
- `goal_key` (строка; уникально в пределах шаблона)
- `type` (строка; например `action_once`, `safety_fund_months`, `passive_income_monthly_min`, `cash_balance_min`, …)
- `title` (строка)
- `order_index` (int; порядок в chain)
- `enabled` (bool)
- `required` (bool)
- `requires_mechanics` (text[] или JSONB-массив строк)
- `params` (JSONB) — типо-специфичные параметры:
  - `action_once`: `{ action: "salary_claimed" | ... }`
  - `safety_fund_months`: `{ months_multiplier: 3 | 6 }`
  - `passive_income_monthly_min`: `{ min_monthly: 15000 }`
  - `cash_balance_min`: `{ min_cash: 10000000 }`

Индексы/уникальность:
- unique (`template_key`, `goal_key`)
- index (`template_key`, `order_index`)

### 2) Источник истины в backend
В backend заменить чтение целей из `game_starter_templates.victory_config_json` на загрузку из `victory_goals` по `template_key`.

При этом верхний “конфиг победы” можно оставить как сейчас (в JSON) на первом шаге:
- `min_period_index_for_victory`
- `required_goals_met`
- `progression_mode`
- `playtest_mode`

Позже (не в MVP) можно вынести верхний конфиг в отдельную таблицу `victory_template_config`.

### 3) Валидация (must-have)
Добавить серверную валидацию при старте приложения / в тестах:
- уникальность `goal_key` в пределах `template_key`
- корректность `order_index` (нет дыр/дубликатов, если договоримся)
- соответствие `params` типу `type` (строгая схема на стороне Python)
- валидация `requires_mechanics`:
  - ключи из известного списка механик,
  - цели не требуют механику, которая никогда не может быть доступна в данном шаблоне (template cap)
- для `action_once`: `action` должен быть поддержан движком

### 4) Миграция данных (dev-first)
Сделать миграцию, которая:
- создаёт таблицу `victory_goals`,
- переносит текущие цели из существующих `victory_config_json` в строки (минимум: 4 шаблона),
- оставляет `victory_config_json` как бэкап/legacy на время перехода (не источник истины).

## Not Doing (and Why)
- Не делаем overrides на уровне профиля — это усложняет поддержку и аналитику; сейчас достаточно `template_key`.
- Не делаем полностью нормализованную схему “по таблице на тип цели” — дорого в разработке и будет мешать эволюции типов.
- Не делаем live-редактирование без деплоя в MVP — админка позже; сейчас достаточно миграций/seed + строгих проверок.
