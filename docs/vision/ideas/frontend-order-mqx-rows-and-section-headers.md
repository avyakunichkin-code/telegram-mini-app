# Порядок на фронте: границы + MQX (Rows + Section Header)

## Problem Statement
**Как сделать так, чтобы фронтенд развивался предсказуемо (структура) и выглядел ровно (UI), не делая большого рефактора, в соло‑режиме разработки.**

Фокус внедрения (первые точки приложения):
- **Dashboard**
- **Капитал** (вкладка `finance`)

---

## Recommended Direction
Берём связку **границы папок / legacy‑quarantine + MQX-first** и применяем её дисциплинированно.

### Структура (границы без боли)
- Всё новое и существенное — в `screens/…` (route/flow-level) и/или `components/mqx/` (видимые паттерны).
- `components/` остаётся legacy‑зоной и **не расширяется** для новых экранов/паттернов (канон — `frontend-react/ARCHITECTURE.md`).
- Любые переносы из legacy — только **touch-it move-it** (по касанию).

### UI (MQX-first для 2 ключевых паттернов)
Целевые паттерны для унификации:
- **B: “карточки/строки”** (list items/rows для списков на Dashboard и в Капитале)
- **C: “заголовок секции + actions”** (section header с иконками/кнопками/вторичными действиями)

Правило: новые/заметно меняющие эти паттерны штуки идут по циклу MQX:
`design-lab (2–5 вариантов) → явное утверждение → MQX → витрина #/dev/mqx → внедрение (1 экран/сценарий за PR)`
см. `frontend-react/src/components/mqx/DESIGN_WORKFLOW.md`.

  - Как проверить: после внедрения вы начинаете повторно использовать `mqx/` вместо копипасты из legacy.
- [ ] **Витрина `#/dev/mqx` используется как единственный каталог**, а не “папка, про которую забыли”.
  - Как проверить: при следующей UI-задаче сначала смотрите витрину.

## MVP Scope (1–2 недели)
### MVP-структура (границы + карантин)
- Закрепить правило “новое — только `screens/` и/или `mqx/`”, legacy — только “по касанию”.
- Сделать 1 маленькое “доказательство” правила: при ближайшем изменении вынести один touched flow-level кусок в `screens/...`, оставив thin-wrapper в legacy (без массовых переносов).

### MVP-MQX: Rows + Section Header (сразу качественно)
Цель — сделать “ядро паттерна” так, чтобы **дальше не переписывать по 100 раз**, а только расширять.

#### Паттерн B — Row/Card для списков
- Design-lab round: 2–4 варианта строки/карточки списка (единые тестовые данные).
- После утверждения:
  - MQX-компонент(ы) в `components/mqx/…`
  - стили `mqx-*` в `src/styles/…` (без inline-hex в компонентах)
  - секция в витрине `#/dev/mqx`
- Внедрение: **один экран / один сценарий за PR** (начать с Dashboard или Капитала).

#### Паттерн C — Section Header + Actions
- Design-lab round: 2–4 варианта заголовка секции (title/subtitle, action button/icon, optional counter/badge).
- После утверждения: аналогично (MQX + стили + витрина), затем внедрение в одной точке.

---

## Состояния (делаем сразу, чтобы не переделывать)
Принцип: паттерны должны быть пригодны “как кирпичики” для реальных экранов, поэтому закладываем полный набор состояний с MVP.

Обязательные состояния для обоих паттернов (Row/Card и Section Header):
- **Loading**: skeleton/placeholder (на уровне секции и/или списка).
- **Empty**: “ничего нет” с аккуратным CTA/подсказкой (где уместно).
- **Error**: сообщение + действие “повторить” (если есть запрос) или “что делать”.
- **Disabled / Read-only**: когда механика ещё не доступна / действие временно недоступно.
- **Warning / Danger** (по смыслу): просрочки, риски, негативные значения (в Капитале актуально).

Необязательные, но желательно заложить API/слоты:
- **Trailing action** (иконка/кнопка), **secondary text**, **badge/counter**, **right-aligned value** (денежные значения), **leading icon/illustration**.

---

## Not Doing (and Why)
- **Не делаем массовый переезд `*Premium.jsx`** — только touch-it move-it (соло-скорость важнее).
- **Не делаем редизайн** — только унификация паттернов B и C.
- **Не берём >2 паттернов за итерацию** — иначе расползётся в бесконечную полировку.
- **Не улучшаем legacy “для красоты”** — только если трогаем по задаче.

## Приложение A — все варианты идей (для будущих итераций)
Чтобы не потерять направления, которые “позже станут актуальны”.

### 1) “Границы папок + карантин legacy” как продуктовая фича
- Усилить правило “новое — только `screens/` и/или `components/mqx/`”.
- `components/*Premium.jsx` и `components/*Section.jsx` держать в явном quarantine.

### 2) “Touch-it move-it” миграция вкладок игры → `screens/game/`
- Постепенно выносить куски из `*Premium.jsx` при касании, оставляя thin-wrapper.

### 3) “MQX-first” для визуальной консистентности
- Выбирать 3–5 самых заметных паттернов разнобоя и проводить их через design-lab → MQX → витрина → внедрение.

### 4) Единые дизайн-токены на CSS-уровне (без редизайна)
- Закрепить один источник правды для spacing/radius/shadows/typography/z-index/colors.
- Новое — только через токены; legacy — по касанию.

### 5) “Каталог компонентов/паттернов” как встроенная документация
- Развивать витрину `#/dev/mqx` как “единственное место, где видно как правильно”.

### 6) “Контракт UI ↔ API” как слой порядка
- Единообразная обработка ошибок/ответов в `src/api/*`, чтобы экраны не плодили разные паттерны.

### 7) “Гигиена репозитория” (чтобы порядок не ломался сам)
- Артефакты сборки/кэши не попадают в git; короткая процедура “как чистить” и/или автоматическая проверка в CI/хуках.

## Приложение B — MQX-first: 5 паттернов (кандидаты для выравнивания)
Смысл: выбрать ограниченный набор “повторяющихся кирпичиков” и прогонять их через design-lab → утверждение → MQX → `#/dev/mqx` → внедрение.

Приоритет — **Dashboard** и **Капитал**.

1) **Section Header + Actions (C)**
   - Заголовок секции: kicker/title/subtitle, meta (например “N поз.”), trailing action (кнопка/иконка).
   - Design-lab: `design-lab/section-headers/` (варианты A–D).

2) **List Row / Compact Position Row (B)**
   - Универсальная строка списка для позиций и каталогов: title/subtitle/metrics/trailing actions.
   - База: `MqxFinListRow` + `MqxRowAction` + `MetricsRow`.

3) **Empty / Error / Loading states**
   - Единый UI для: пусто, ошибка, загрузка (скелеты) — чтобы не плодить “Загрузка…” и разные пустые тексты.
   - Должно работать внутри секций Dashboard и Капитала.

4) **Card Header (kicker/title/sub) как единый канон**
   - Вынести “карточную шапку” в один MQX-компонент и использовать везде, где сейчас руками `mqx-card__kicker/title/sub`.
   - База: `MqxCardHeader` (расширить/унифицировать usage).

5) **Spacing / Stack / Dividers**
   - Единая вертикальная сетка: отступы между блоками, разделители, контейнеры.
   - База: `MqxDashStack` + `MqxDivider` (и их аналоги для капитала).
