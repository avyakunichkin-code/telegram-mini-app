# Идея: механика «Расходы» (жизнеобеспечение)

## Problem Statement (How Might We)

**Как встроить в ТВОЙ ХОД полноценный слой «расходы на жизнь» (еда, жильё, одежда, связь, здоровье, досуг и т.д.), чтобы игрок понимал, куда уходит зарплата каждый период, мог учиться управлять burn rate, а экономика, события, победа и Plan Mode опирались на одну модель — без дыры между «доходом» и «обязательствами по кредитам»?**

Сейчас в Game Mode есть **одна скрытая цифра** (`base_monthly_lifestyle_expense` + дельта событий), списываемая в конце периода. Это **не отражает продуктовую реальность**: человек тратит на **категории жизнеобеспечения**, часть из них меняется событиями, часть зашита в шаблон старта, часть в будущем — вводится в Plan Mode вручную.

---

## Phase 1 — Diverge: варианты модели

| # | Вариант | Суть | Плюсы | Минусы |
|---|---------|------|-------|--------|
| **V1** | Одна строка «Жизнь» (как сейчас) | Агрегат на профиле | Быстро, уже в коде | Нет педагогики, нет контента по категориям, дыра в логике |
| **V2** | Агрегат снаружи, **категории внутри** | UI одна цифра; БД — строки по категориям | Game UX простой; модель богатая | Нужна синхронизация sum = total |
| **V3** | **Категории видны игроку** | 5–8 статей на дашборде/экране «Расходы» | Максимальная прозрачность | Перегруз Game; ближе к Plan |
| **V4** | Обязательные + **дискреционные** | Два уровня: «нельзя не платить» vs «можно ужать» | Хорошо для обучения бюджету | Сложнее баланс и события |
| **V5** | **Счета-коммуналки** | Отдельные due-даты внутри периода | Реализм | Ломает простой цикл «конец месяца» |
| **V6** | Расходы = **микро-обязательства** | Каждая статья как `FinanceLiability` без тела | Переиспользование кода | Путаница с кредитами; семантика ломается |
| **V7** | **Гибрид V2 + V4** (рекомендация) | Каталог категорий; строки бюджета на профиле; tier must/discretionary; Game UI — свёрнуто → разворот | Масштаб на Plan; контент; одна формула | Большой эпик, нужны фазы |

### Линзы (кратко)

- **Inversion:** что если расходы **не списывать** автоматически, а требовать «оплатить месяц жизни»? → отложено (evolution: ручная оплата — горизонт).
- **Audience shift:** Plan — полный редактор статей; Game — шаблон + события, без ручного ввода на старте.
- **10x:** сезонность, инфляция, региональные коэффициенты — **не v1**.
- **Expert (семейный бюджет):** статьи должны сходиться в **месячный burn** и в **% от дохода**.

## Phase 2 — Converge: рекомендуемое направление

### Выбор: **V7 — структурированный бюджет жизнеобеспечения**

1. **Доменная сущность:** «статья расходов» (expense line) — категория, сумма/мес, источник, срок, тип (обязательная / гибкая).
2. **Итог периода:** сумма всех активных статей → списание с `cash` (как сейчас lifestyle, но с **breakdown по категориям**).
3. **Game Mode UX:** по умолчанию **одна цифра** на дашборде + экран «Расходы» с расшифровкой (V2 снаружи, V3 внутри экрана).
4. **Шаблоны:** `expense_budget` в blueprint (разбивка по категориям), сумма = бывший `base_monthly_lifestyle_expense`.
5. **События:** эффекты на **категорию** или на **весь burn**; временные статьи с `expires_period_index`.
6. **Граница с долгами:** **ипотека/кредит** — `FinanceLiability`; **аренда** без тела долга — статья `housing` в расходах. Шаблон явно задаёт, что куда.
7. **Plan Mode (позже):** те же таблицы + CRUD статей игроком; Game только read + события.

### Stress-test

| Критерий | Оценка |
|----------|--------|
| **Ценность** | Закрывает главную дыру симулятора «зарплата − кредиты ≠ жизнь» |
| **Реализуемость** | Большой, но режется на 4–5 релизов; E1.0 можно без UI-категорий |
| **Дифференциация** | Отличает MQ от «таблицы долгов»; стыкуется с финграмотностью |

### Скрытые допущения (нужно валидировать)

| Допущение | Риск если ложно | Как проверить |
|-----------|-----------------|---------------|
| Игроку в Game хватит **одной цифры + экран деталей** | Перегруз или скука | Плейтест 5–7 человек |
| **5–7 категорий** достаточно | Слишком грубо/мелко | Контент-ревью + сравнение с семейными бюджетами |
| Автосписание в конце периода ок | Хотят «оплачивать» сами | Опрос; отложить manual pay |
| Аренда vs ипотека в шаблоне **не путают** | Двойной учёт жилья | Чеклист шаблонов + тесты |
| Burn входит в **monthly_reference_expense** | Достижения/подушка «ломаются» | Регрессия achievement_engine |

### Что убиваем идею

- Дублирование жилья в liability + housing expense в одном шаблоне.
- Два источника правды (поле `delta` на профиле и строки adjustments без sync).
- Включение расходов в `net_monthly_cashflow` без UX — игрок не поймёт минус.

## Phase 3 — MVP Scope эпика (волны)

### Волна A — «Логическая правда» (backend, без богатого UI)

- Каталог категорий (сиды).
- Строки бюджета на профиле; расчёт total; period end; overview breakdown.
- Миграция шаблонов: разложить `base_monthly_lifestyle_expense` на категории.
- Legacy поля — алиас total.

### Волна B — «Игрок видит»

- Экран / блок «Расходы», дашборд, закрытие периода.
- `total_monthly_outflow`, копирайт cashflow.

### Волна C — «Контент и цели»

- События по категориям; victory `expense_to_income_ratio`; достижения «расходы ≤ N% дохода».

### Волна D — «Plan»

- Редактирование статей; префилл из снимка; отличия UX Game/Plan.

## Not Doing (v1 эпика)

- Ручная оплата каждой статьи в Game Mode.
- Несколько due-date внутри одного периода (коммуналки по числам).
- Инфляция / индексация статей без события.
- Налоги как отдельный модуль (связать с backlog «налоги»).
- Полный паритет Plan Mode в первом релизе E1.

## Принятые решения (2026-05-19)

| Вопрос | Решение |
|--------|---------|
| **Жильё на старте** | **Минимальный `housing`** в базовом шаблоне; рост доли жилья по шаблонам сложности |
| **Нехватка cash** | Списание burn **в минус** (как maintenance/lifestyle сейчас) |
| **Game UI** | **Дашборд:** одна цифра burn (уже есть плитка «Расходы»). **Финансы:** полноценный блок — **3–5 топ-категорий** + детализация |

## Open Questions (осталось)

1. **Дискреционные статьи:** снижение только событиями или кнопка «экономлю» в Game? → **v1: только события**
2. **Категории в Plan:** фиксированный каталог или свои названия? → **v1: каталог; кастом в D**
