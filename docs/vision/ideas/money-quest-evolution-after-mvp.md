**Статус документа:** **§§ Problem … Снимок реализации** — исторический контекст и старые формулировки (easy/hard); **актуальная целевая концепция, Q&A и пошаровый план** — в **[части II](#часть-ii--полная-концепция-после-обсуждения-qa-и-план-работ)** внизу файла. ## Problem Statement

Как усилить игровую и учебную ценность ТВОЙ ХОД поверх уже реализованного периодного цикла, шаблонов активов/долгов, событий, инвестиций и страховок — без потери ясности правил и одного источника правды на сервере?

---

## Recommended Direction

Опорная линия: **сначала связность систем (события + страховки/типы активов)** с узким набором эффектов и явными подсказками в UI, затем **точечное усложнение «hard»** (отдельный режим или флаги профиля) с 1–2 правилами за релиз (например штраф к просрочке или простой налоговый слой), а не большой пакет сразу.

Параллельно — **мета-цели** (повторяемая победа, «чистые» серии, необязательные челленджи), опирающиеся на уже существующие `clean_period_streak` и аналитику timeseries, чтобы дать смысл играть дольше одной победной траектории.

---

## Key Assumptions to Validate

- [ ] Игроки понимают разницу cash / подушка / обязательства без отдельного туториала — проверка: короткий опрос или метрика «первый провал по просрочке после N периодов».
- [ ] До трёх событий за период не перегружают сессию — A/B или качественные интервью.
- [ ] Расширение эффектов событий (страховка снижает урон) не ломает баланс победы MVP — регрессионные сценарии в тестах/фикстурах на `process_period_end`.

---

## MVP Scope (следующий «слой» после текущего кода)

- Каталог событий: минимум **одна** механика «страховка меняет исход/сумму» + отображение в карточке события.
- Контракт API: расширение только там, где нужно (например метаданные выбора или флаги в overview), с обновлением `frontend-react/src/api.js` и экранов событий.
- Документация: обновить [`foundation/SPEC_PRODUCT.md`](../foundation/SPEC_PRODUCT.md) под новый слой; feature spec — [`SPEC_game-plan.md`](../../specs/features/SPEC_game-plan.md).

## Not Doing (and Why)

- Полный налоговый симулятор и все виды кредитов в одном релизе — слишком высокая стоимость объяснения и баланса.
- Alembic и тяжёлая миграционная история — против текущего осознанного выбора проекта; крупные схемы только через чёткие SQL-миграции по шагам.
- Обязательные сценарии событий (`mandatory`) до того, как стабилизированы базовые ветки эффектов — риск мёртвых состояний и блокировок UX.

## Снимок реализации (якорь)

- **Цикл:** период с таймером → действия (зарплата по кнопке, подушка, активы/долги из шаблонов, события, инвестиции, страховки) → `process_period_end` в `backend/app/game_period.py` (обслуживание активов, доход активов, долги и просрочка, премии страховок, инвестиции, поражение при трёх отрицательных периодах подряд, снимки, новые события).
- **Победа MVP:** в `GET /api/finance/overview` — подушка ≥ 3× ежемесячные обязательства, нет просрочки, неотрицательный net cashflow; **дополнительно** `win_reached` только с `period_index >= 7` (первые шесть периодов без победы).
- **Фронт:** `StartMenuScreen` → **`NewProfileKindScreen`** → **`GameTemplatePickScreen`** → `GameScreen` (ветка Game); **`BaseParamsScreen`** зарезервирован под Plan; финансы, аналитика, премиум-дашборд; события оверлеем.

## Часть II — Полная концепция (после обсуждения), Q&A и план работ

Ниже зафиксировано **целевое видение**, согласованное в диалоге (май 2026), и **план доработок по слоям**. Верхние разделы документа (про easy/hard как пару сложности) **частично устарели**: целевая модель — **два режима сохранения** (`Game` / `Plan`) и шаблоны сложности в Game; переименование и вычистка `light` / `hardcore` в коде — только по отдельному согласованному PR.

### II.1. Полная концепция (сводка)

#### Разделение «игра» и «инструмент планирования»

- При старте **новой игры** пользователь выбирает **режим сохранения** (immutable на весь жизненный цикл профиля):
  - **Game Mode** — приоритет **игры**: без ручного заполнения «реальной жизни» на входе; старт с **игровых шаблонов**; расходы в UI **агрегированно** (одна линия «жизнь» + изменения из событий), педагогика через уровни, события, достижения.
  - **Plan Mode** — приоритет **планирования**: полное ручное заполнение, статьи расходов как в семейном бюджете, редактируемые параметры кредитов/ипотеки/страховок и т.д.
- **Смена режима у существующего сохранения невозможна**; в списке сохранений — лейблы `game` / `plan` (цветом). Для **Game** дополнительно отображать **сложность** (ранг шаблона / difficulty из шаблона).

#### Game Mode — онбординг и шаблоны

- **Первый вход в Game Mode:** сразу старт с **одним базовым шаблоном** (минимальный доход, «примитивный» персонаж, без активов и подушки, только зарплата) — без лишних настроек.
- **Повторное «Новая игра / Game»:** выбор из **4–5 шаблонов** нарастающей сложности (типизация: базовый; специалист со средней зарплатой и повышенными базовыми расходами + сюжетный контекст; семья + ипотека; предприниматель с высоким доходом и **волатильностью**, ипотека, автокредит, кредит и кредитная карта, **критический** уровень расходов и т.п.).
- Шаблон задаёт **стартовые параметры и цели победы** (см. ниже) — **одна сущность** для баланса и контента.

#### Прогрессия персонажа и контент

- **Уровни персонажа:** линейная цепочка разблокировки механик допустима; **порог XP между уровнями** — нелинейно растущий (классическая «кривая сложности»), настраивается данными.
- **Вторая ось — достижения:** многоуровневые (например 3–5 тиров на достижение), пример — подушка к кратности «месячных расходов»; **за тиры начисляется XP** в общий уровень.
- **События:** привязка к **уровню события** (на первом этапе синхронно с уровнем персонажа); пул выпадения из диапазона **от текущего уровня событий до двух уровней ниже**; более сложные события открываются с ростом уровня и доступных механик (например события про недвижимость — после разблокировки покупки).
- **Обязательные события** (с 4–5 уровня и выше по задумке): **смесь** — часть **блокирует** завершение периода / ключевой шаг, часть даёт **только финансовые** последствия без жёсткого блока; предикаты (есть авто → может ДТП; страховка меняет исход; временные расходы вроде «такси 3 месяца»).
- **Опыт за «платные» выборы в событиях:** **фиксированный XP** на уровне определения выбора в данных (БД / JSON с единым применителем на бекенде).

#### Расходы (Game)

- **Базовые месячные расходы «жизни»** — фиксированы в **шаблоне** (не отдельные активы).
- **Дельта расходов** — из **разрешённых событий** (собака, телефон и т.д.; телефон не актив, только рост регулярных расходов). Новые сценарии добавляются **через БД**; схема должна поддерживать эффекты `monthly_expense_delta`, временные модификаторы (по желанию — строки с `expires_period`).
- **Повторяемость событий:** флаг / политика — **безлимитные повторы**, **один раз на профиль** или **максимум N раз**.

#### Победа (Game Mode — целевая модель)

- **Условие:** выполнить **M из N целей** (обсуждалось **3 из 5**); конкретный **набор целей, пороги и M** хранятся в **шаблоне**.
- Примеры целей (пул, настраивается):
  1. Достижение **последнего уровня развития** (число уровней не зафиксировано жёстко в «10»; ориентир — **9 уровней** с плавным открытием механик: зарплата → подушка → лёгкие события → депозит → кредиты → страховки → крупные активы/ипотека → кредитные карты → капитал / инвест. недвижимость).
  2. Подушка **≥ 6×** от опорных **месячных расходов** (база шаблона + дельты из событий).
  3. Средний **чистый денежный поток** за последние **K = 6** периодов **≥ порога** от **стартовой зарплаты** (коэффициент задаётся шаблоном, для «лёгкого» обсуждалось **×5**).
  4. **Кэш на счёте** ≥ целевого порога (глобальная цифра типа «25 млн» заменяется на **порог из шаблона** для гибкости).
  5. Покупка **инвестиционной недвижимости** (или другая финальная механика из шаблона).
- **Ранний выигрыш запрещён:** в первые **6 игровых периодов** победа **невозможна по дизайну и технике** — в реализации MVP на бекенде: `win_reached` только при `period_index >= 7` (`MIN_PERIOD_INDEX_FOR_WIN` в `backend/app/routers/finance.py`). При введении шаблонной победы то же правило переносится в общий движок условий победы (шаблон может задать `min_period_index_for_victory`, по умолчанию 7).

#### Cashflow (цель 3)

- **Определение:** средний **чистый** денежный поток за последние **6** завершённых/учтённых периодов (одна формула на сервере, одно поле в overview для UI и победы).

#### Plan Mode и связь с Game

- **Нет переноса прогресса** между профилями разных режимов «как конвертация».
- При создании **новой игры Plan** — опция **подтянуть стартовые данные из выбранного сохранения** (любого режима по смыслу «у меня уже есть снимок»): подтягивается **только стартовый снимок** (балансы, активы, обязательства, агрегаты расходов **на момент старта** той игры), чтобы не заполнять вручную заново для долгосрочного моделирования.
- Для этого у сохранения должна быть **неизменяемая запись стартовых параметров** (например `starter_params_json` при создании профиля).

#### Горизонт: ручная оплата расходов

- В концепции зафиксировано **будущее усложнение:** ручное подтверждение оплат по категориям; пропуск → **задолженность**. Для **базового Game-шаблона и ближайших релизов** — **полностью автоматические** списания, как сейчас.

#### Именование в коде (миграция)

- Текущие **`light` / `hardcore`** на профиле и в событиях **не входят** в новую концепцию; замена на **`save_kind`: `game` | `plan`** и параметры шаблона (**difficulty** и пр.). Удаление старых полей/веток — **только по согласованному списку** (см. план), чтобы не сломать прод и миграции БД.

### II.2. Вопросы и ответы (по переписке)

| # | Вопрос / тема | Ответ / решение |
|---|----------------|-----------------|
| 1 | Как разделить симулятор и фин. грамотность? | Два режима при новой игре: **Game Mode** (игра, шаблоны, без ручного ввода на старте) и **Plan Mode** (планирование, статьи расходов, ручные параметры). |
| 2 | Как стартует Game в первый раз? | Сразу **базовый шаблон**, без выбора. |
| 3 | Повторная новая игра в Game? | **4–5 шаблонов** разной сложности и лора. |
| 4 | Как устроить прогрессию «интереснее линейного списка механик»? | Линейная цепочка механик + **нелинейная кривая XP**; **вторая ось — многоуровневые достижения** (3–5 тиров), XP за достижения идёт в уровень. |
| 5 | Обязательные события — всегда блок периода? | **Смесь:** часть с блоком конца периода, часть только с финансовыми последствиями. |
| 6 | Как задать cashflow для цели победы? | **Средний чистый поток за последние 6 периодов**; порог от стартовой зарплаты — **в шаблоне** (гибко). |
| 7 | Цель «много кэша» для всех шаблонов? | **Порог кэша в шаблоне** (и при необходимости другие пороги). |
| 8 | Откуда брать набор целей победы? | **Целиком из шаблона** (какие цели активны, сколько нужно выполнить из пула). |
| 9 | Повторяемость событий? | Флаг: **повторяемые / один раз / максимум N** на профиль. |
| 10 | Ручная оплата расходов? | **В концепции на будущее**; сейчас **авто** для базового опыта. |
| 11 | Plan: откуда брать данные при «новая игра из сохранения»? | Только **стартовый снимок** того сохранения. |
| 12 | Список сохранений? | Лейблы **game / plan** + для game ещё **difficulty**. |
| 13 | Первые периоды и победа? | Победа **недоступна** в периодах **1–6**; **технически** — `period_index >= 7` для `win_reached` в текущем MVP. |
| 14 | Удалять ли light/hard из кода сразу? | Замена на **`save_kind`** и шаблоны выполнена в **G1** ([ADR-001](../decisions/ADR-001-save-kind-remove-light-hardcore.md)); дальнейшие флаги режимов — только через ADR/спеку. |
| 15 | Конфликт имён `mode` на `GameProfile` vs события? | Новая концепция: **отдельное поле** для game/plan; не смешивать со старым `light/hardcore`. |
| 16 | Сколько уровней развития вместо «10»? | Ориентир **9 уровней** с плавным открытием механик (до капитала / инвест. недвижимости). |
| 17 | Какой диапазон уровней у выпадающих событий? | События уровня **от текущего до двух уровней ниже** (лёгкие не пропадают сразу). |
| 18 | XP за платные выборы в событиях? | **Фиксированный** XP в данных выбора; уровень событий на первом этапе **синхронен** с уровнем персонажа. |
| 19 | Перенос сохранения между режимами? | **Нет**; Plan может только **префиллить старт** из `starter_params_json` другого сохранения. |
| 20 | Типы шаблонов Game (пример)? | Базовый; специалист + расходы + контекст «до свадьбы»; семья + ипотека; предприниматель + риски + стресс по обязательствам. |
