# MVP 1.1 — закрытые ответы и направления доработки

Конспект решений из продуктового диалога + **упакованные идеи** для следующего этапа (**spec → plan → задачи**). ## 1. Зафиксированные ответы (чеклист)

| Тема | Решение |
|------|---------|
| ЦА | 14–35, новички в финграмотности; через игры ищут **пользу** и более «интеллектуальный» опыт |
| Сессия **сейчас** | ~**1–3 мин** типичный заход |
| Сессия **цель роста** | к **10–15 мин** «плотного» захода за счёт контента и меты, без принуждения сидеть дольше периода |
| Кредиты | обычный **заём / кредит** — ок; **микрозаймы**, **казино/азарт наградой**, хищные «до зарплаты» — **нет** ([§3 TARGET](../../foundation/TARGET_PLAYER_AND_SESSION.md)) |
| Последствия ошибок | **жёстче ощущаемые**, с **понятной** обратной связью ([§4 TARGET](../../foundation/TARGET_PLAYER_AND_SESSION.md)) |
| Возврат в приложение | **цель победы** и **прогресс по уровню / прокачке** видимы как главные крючки |
| Тип игроков | и **одна кампания до победы**, и **экспериментаторы**/новые партии — оба поддерживаем ([§6 TARGET](../../foundation/TARGET_PLAYER_AND_SESSION.md)) |
| Тексты событий | **короткий заголовок** + **немного текста** смысла; термины — сноска/мелкая строка/краткая врезка ([§7 TARGET](../../foundation/TARGET_PLAYER_AND_SESSION.md)) |

## 2. Формулировка проблемы («How Might We»)

Как при **заходах 1–3 минуты** держать **мотивацию кампании и экспериментаторов**, если «мясо» игры уже сейчас в **событиях и месяцах**, а недостает **понятной эскалации сложности, разнообразия решений и видимого роста** — при наших табу контента и требовании **жёстких но читаемых** последствий?

---

## 3. Стратегический вывод для 1.1

Максимальный эффект на **механику и увлечённость на единицу работы** даёт связка:

**события (вариативность + текст) × привязка к прогрессу персонажа × явная мета-панель победы/уровня в UI.**

Расширение справочников активов/долгов — **вторично**, точечно **только если** нужно условием события или новому шаблону.

Долгая цель **victory M из N** и полный «движок победы» из шаблона остаются **следующей крупной волной** после или параллельно «тонкому» 1.1, если решите разрезать scope (см. не-делать ниже).

## 4. Направления доработки (идеи → будущие эпики)

### A. События по уровням персонажа

- Поле **`event_level`** (или эквивалент) у определений событий; правило набора периода: например диапазон «текущий уровень персонажа … −2», с защитой от пустых периодов.
- Отдельные бакеты для **tutorial / low / mid** без ломания `save_kind` filter.
- **Контент:** 10–20 новых карточек или переработка существующих под шаблон **заголовок + тело** и табу TARGET.

### B. Видимая прокачка (тонкий минимум)

- Источники **XP**: выбор в событиях (конфиг), **чистые периоды**, ключевые «учебные» действия при необходимости.
- **Пороги уровня** в конфиге или простой таблице; на уровне N — допускаются события уровня N (связка с блоком A).
- UI: блок **«Уровень N»** и прогресс к N+1 **на главной** или hero (без второй параллельной «глобальной геймификации» без смысла).

### C. Последствия «жёстче, но честно»

- Усиление **понятных** санкций: штрафы к «жизни», к подушке, к просроку — там, где сейчас «пролетело мимо игрока».
- Никаких **скрытых наказаний** без цифры в итоге периода или тоста (в духе TARGET §4).

### D. Цель победы и «зачем ещё заходить»

- До полного victory v2: **явный прогресс к текущей победе MVP** (подушка, просрок, поток — что уже есть в `overview`) в одном компактном блоке для **архетипа кампании**.
- Для экспериментаторов — **ясное различие шаблонов** (ключ, сложность, один экран «чем этот отличается» при выборе).

### E. Сущности (точечно)

- Новые активы/обязательства/страховки **только** если закрывает сценарий событий или пробел шаблонов MVP 1.1; без «расширения зоопарка ради галочки».

---

## 5. Предложение границ MVP 1.1

**Входит (реалистичный вертикальный срез):**

1. Спека + данные **событие ↔ уровень** + фильтр в коде набора событий.
2. Контент **первой волны** под новые правила и копирайт TARGET.
3. **XP и уровень** минимальной связкой с событиями/периодом + элемент UI на главной.
4. Одна-две механики **более ощутимых последствий** с UX-объяснением, согласованные с циклом периода.

**Опционально в той же волне или сразу следом:**

- Расширение пула событий с **политикой повторов** (простое: «один раз на профиль» / «повторимо»).
- Заготовки под **страховка ↔ тип события** (узкий первый слой из бэклога).

**Не входит в 1.1 без отдельного решения:**

- Полный UI **Plan** и префилл.
- **Victory M из N** из шаблона как единственный критерий победы (поля среднего cashflow в overview можно показывать, но смену правила победы — отдельный эпик).
- Массовое раздувание справочников сущностей без привязки к событиям.
