# Design-lab всегда актуален: Canon Manifest + Page Parity

## Problem Statement
Как сделать так, чтобы `design-lab` оставался **актуальным относительно prod** “по умолчанию”:

- чтобы **любой агент/разработчик** мог понять текущий канон без контекста чата;
- чтобы утверждённые блоки **не деградировали на уровне страницы** (когда “вариант блока ок”, а в реальной композиции страницы выглядит плохо);
- чтобы обновления канона были **не ручной дисциплиной**, а **проверяемым контрактом** (CI/guardrails).

## Recommended Direction
Разделить канон на **два уровня правды** и добавить **машинно‑проверяемый манифест**.

### 1) Block Canon (канон блоков)
Для отдельных блоков/элементов (например: needs summary, goal collapsed path, finance chips, period actions) каноничны:

- **структура и инварианты** (что обязательно/что запрещено),
- **ключевые состояния** (empty/loading/error, если в prod уже есть),
- **порядок внутри блока** (например, порядок чипов/строк/метрик),
- **критичные UX‑решения** (например: “needs всегда открыто”, “goal collapsed = stepper без текста ‘Шаг N из M’”).

Здесь **не обязателен** пиксель‑перфект в вакууме — потому что истинная проверка происходит на уровне страницы.

### 2) Page Canon / Prod Parity (канон страниц)
Для 1–2 “общих страниц” (начать с Dashboard, затем Finance) канонично:

- **визуальное соответствие** prod (в рамках разумного: композиция, сетка, плотности, заголовки, разделители),
- **порядок блоков на странице**,
- **ключевые состояния страницы** (если уже есть в prod).

### 3) Canon Manifest (машинный канон)
Для каждой темы/страницы добавляем `canon.manifest.json` в `design-lab/<theme>/`:

- **порядок блоков** (page layout order),
- ссылки на **источники правды** (какие round’ы считаются canonical),
- статусы: `approved` / `superseded` / `archive`,
- список **обязательных состояний** (для page parity и/или block canon).

Главный принцип: **манифест — контракт**, а `APPROVED.md` — человекочитаемая фиксация решений.

### 4) Page parity не собираем “склейкой раундов”
Запретить (процессно и технически) “ручную склейку CSS/HTML из нескольких round’ов” для page-round, потому что она:

- создаёт глобальные конфликты стилей,
- ломает футер/лейаут “не по делу”,
- даёт нестабильные результаты (как показала текущая сессия).

Вместо этого:

- page-round **генерируется** из манифеста (скриптом), либо
- собирается из **единого page-shell** + **встраиваемых блоков** с изоляцией (iframe или строгий контракт встраивания без глобальных стилей).

### 5) Enforcement (guardrails)
Добавить/закрепить проверки:

- **Структурный чек**: порядок блоков в `frontend-react` (например, `DashboardPremium.jsx`) должен совпадать с `design-lab/dashboard/canon.manifest.json`.
- **Canon Sync чек**: если менялись MQX/экран (или другие UI‑паттерны), а `design-lab/**` не обновлён — PR красный (существующий подход развить до “manifest‑driven”).

Результат: канон обновляется не “по памяти”, а как часть изменения, и это проверяется автоматически.

## Key Assumptions to Validate
- [ ] **Сходимость “Block → Page”**: если блок утверждён, но “на странице плохо”, то это считается дефектом процесса, а не “ну бывает”; корректировка происходит через новый round, а не через ручные костыли в page-round.
- [ ] **Манифест достаточно выразителен**: порядок блоков + источники canonical round’ов + статусы покрывают 80% причин рассинхрона.
- [ ] **Изоляция стилей обязательна**: page parity без изоляции (склейка CSS) будет повторно ломаться при росте количества round’ов.

## MVP Scope
Минимальный набор, чтобы система начала держать `design-lab` актуальным и масштабировалась на Finance.

### Dashboard (сразу)
- `design-lab/dashboard/canon.manifest.json`
- `design-lab/dashboard/PARITY.md` (как проверять “prod parity” и что именно parity гарантирует)
- `scripts/check-dashboard-canon.mjs` (сверка порядка блоков в prod vs manifest)
- `scripts/build-dashboard-page-round.mjs` (генерация page-round без “ручной склейки”)
- Обновить `design-lab/dashboard/APPROVED.md` так, чтобы:
  - порядок блоков был однозначен,
  - инварианты по заголовкам/needs/goal/actions были зафиксированы,
  - round’ы со статусом `superseded` не считались источником правды.

### Finance (заложить шаблон, активировать по мере боли)
- Завести `design-lab/finance/canon.manifest.json`
- Сделать чек порядка блоков для главных секций Finance (визуальный parity включить позже, когда начнутся заметные изменения).

## Not Doing (and Why)
- **Не делаем auto-export из React в design-lab на MVP**: дорого и рискованно; достаточно manifest + порядок + parity‑страницы.
- **Не включаем pixel-diff регрессию сразу**: сначала стабилизируем структуру и сборку page parity; pixel-diff добавится как следующий слой.
- **Не поддерживаем “page-round вручную склеенный из 5 CSS”**: это повторит проблему конфликтов и “ломает соседние блоки”.
- **Не смешиваем superseded варианты в канон**: хранить их можно, но они не участвуют в parity и не считаются источником правды.

- Где лучше хранить manifest‑скрипты: `frontend-react/scripts/` или отдельный `scripts/` в корне репо?
