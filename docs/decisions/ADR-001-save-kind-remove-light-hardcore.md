# ADR-001: Отказ от `light` / `hardcore`, переход на `save_kind` (Game сначала)

## Context

В коде и API сохранений используется пара **`GameProfile.mode`: `light` | `hardcore`**, а каталог событий фильтруется через **`EventDefinition.mode`** (`light` / `hardcore` / `any`). Продуктовая концепция изменилась: различие «лёгкий / хардкорный симулятор» **не является** главным разделением; целевая модель — **тип сохранения `save_kind`: игра (`game`) или план (`plan`)**, старт из **шаблонов**, победа из **`victory_config`**.

Продолжать расширять `light`/`hardcore` технически возможно, но это **фиксирует устаревшую модель** в БД, API и UI (**`DifficultyScreen`** — снят в G1, см. [`SPEC_game-plan`](../specs/features/SPEC_game-plan.md)) и мешает эволюции.

Ограничение по релизу: в **ближайшем эпике** реализуется только **Game Mode** и **один полный шаблон end-to-end**. **Plan Mode** отнесён на **MVP 2.0**.

## Decision

1. **Удалить** из доменной модели профиля семантику сложности **`light` / `hardcore`** как обязательный выбор при создании игры (UI и валидация API).
2. **Ввести** на `GameProfile` поле **`save_kind`** со значениями как минимум **`game`**; значение **`plan`** резервируется под MVP 2.0 (может быть недоступно в API до отдельного эпика).
3. **Миграция данных существующих профилей:** всем активным сохранениям выставить **`save_kind = game`** (текущие пользователи не ведут Plan-сейвы).
4. **Каталог событий:** отказаться от фильтра «сложность профиля» в пользу фильтра, согласованного с **`save_kind`** (и в перспективе — уровнем события). Для MVP с одним Game-шаблоном допустимо:
   - переименовать/заменить значения в `EventDefinition.mode` на **`game` / `plan` / `any`**, **или**
   - ввести отдельное поле совместимости и мигрировать сиды за один проход с бэкендом/фронтом.
   Конкретная схема колонок — в задачах эпика; смысл ADR: **не поддерживать параллельно** «и light/hardcore, и save_kind».
5. **Без dual-read:** не вводить фазу, когда приложение одновременно обязано читать **`mode`** и **`save_kind`** для бизнес-логики. Один согласованный релиз: миграция БД + обновление API + обновление клиента (см. последствия).
6. **Plan Mode (prefill, мастер ввода)** — **вне scope** эпика, покрываемого этим ADR; не блокирует удаление legacy `mode`.

## Consequences

- **Breaking change** для клиентов: тела запросов создания профиля и, возможно, формы списка сохранений меняются; старые клиенты без обновления перестанут быть совместимыми.
- Нужны **SQL-миграция(и)** и обновление **`backend/main.py`** (лёгкие `ALTER`), **сидов событий**, **`frontend-react/src/api.js`**, экранов старта.
- Документация: `CLAUDE.md`, `foundation/SPEC_PRODUCT.md` §0, [`specs/features/SPEC_game-plan.md`](../specs/features/SPEC_game-plan.md).

## Alternatives considered

1. **Dual-read** (`mode` + `save_kind` на переходный период) — отклонено: усложняет код и тесты без выгоды при одном Mini App-клиенте под контролем репозитория.
2. **Оставить `mode` как синоним difficulty внутри Game** — отклонено на уровне ADR: сложность должна приходить из **шаблона** (`difficulty_rank` / ключ шаблона), а не из глобального enum «сложность всей игры» в том же поле, что и legacy hardcore.
