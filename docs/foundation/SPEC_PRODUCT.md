# ТВОЙ ХОД — продуктовая спецификация (идея и текущее состояние)

Документ фиксирует **задумку продукта** и **как она реализована в текущем MVP** (периодный цикл, финансы, события, инвестиции, страховки), с отсылкой к коду и архитектуре. Для онбординга разработчиков см. [`CLAUDE.md`](../CLAUDE.md). **§12** — собранные направления улучшения логики, сущностей, механик и расчётов (не обязательно запланированные в ближайшем PR).

## 0. Дорожная карта концепции (актуально)

Целевое развитие — кратко ниже; **детали, Q&A и план по слоям** только в **[`docs/vision/ideas/money-quest-evolution-after-mvp.md`](../vision/ideas/money-quest-evolution-after-mvp.md) — часть II** (не дублировать здесь):

- два **режима сохранения** (**Game** / **Plan**), без смены режима у существующего профиля;
- **Game:** старт с **шаблонов** (первый запуск — один базовый шаблон; повторная «новая игра» — 4–5 шаблонов сложности), агрегированные «жизненные» расходы + дельты из **событий** в БД, прогрессия уровня и **достижений**, победа **M из N целей** из шаблона (в т.ч. средний чистый cashflow за **6** периодов, пороги кэша/cashflow по шаблону);
- **Plan:** ручное планирование, статьи расходов, опция префилла **только стартового снимка** из выбранного сохранения;
- **победа в первые 6 периодов запрещена** (в коде: `win_reached` только при `period_index >= 7`);
- устаревшая пара **`light` / `hardcore`** заменена на **`save_kind`** и шаблоны — см. [ADR-001](../decisions/ADR-001-save-kind-remove-light-hardcore.md) и [`specs/features/SPEC_game-plan.md`](../specs/features/SPEC_game-plan.md).

Ниже **§1–11** описывают **уже реализованный** цикл и MVP-победу, если не оговорено иное. **§12** — направления улучшения модели (логика, данные, механики, математика), без обязательства ближайшей реализации.

## 1. Позиционирование и ядро идеи

**ТВОЙ ХОД** — **Telegram Mini App**, оформленный как **игра по финансовой грамотности**, а не как сухой курс или таблица. Пользователь **проживает условные месяцы**: ограниченное время периода, деньги на счёте, обязательства, активы, неожиданные ситуации и последствия решений — в **безопасной песочнице**, с правилами, близкими к реальности.

**Педагогический тезис (MVP):** часть ошибок допустима и **имитирует реальное поведение** (например, не нажать «получить зарплату» в этом периоде). Это создаёт учебный стресс и повод думать о дисциплине и подушке. В **Game Mode** тот же приём сохраняется без обязательного «ручного бюджета» на старте.

**Ориентир MVP 1.1+ по последствиям:** ставки ошибок должны ощущаться **жёстче**, чем «всё сгладим за игрока» — при понятной обратной связи (см. [`TARGET_PLAYER_AND_SESSION.md`](TARGET_PLAYER_AND_SESSION.md)).

## 2. Что делает игрок в продукте

Игрок ведёт **игровой профиль** (`GameProfile`) с:

| Смысл | Поля / сущности (упрощённо) |
|--------|------------------------------|
| Операционные деньги | `cash_balance` |
| Подушка безопасности | `safety_fund_balance` |
| Время | `time_state` (play/pause), `period_duration_seconds`, `period_anchor_at`, `period_index` |
| Доход | `FinanceSalary` (зарплата в модели месяца) |
| Активы | `FinanceAsset` (стоимость, обслуживание, доход, kind) |
| Долги | `FinanceLiability` (тело, ставка, платёж, просрочка) |
| Расширения MVP | `InvestmentPosition`, `InsurancePolicy`, события (`EventDefinition` / `EventChoice` / `EventInstance`) |

**Экраны (frontend):** главный дашборд (таймер, зарплата, подушка, переход периода, события), финансы (инвестиции, страховки, активы и долги), аналитика, меню. Навигация: `BottomGameNav`.

## 3. Игровой цикл (core loop)

### 3.1. Период

- Один **период** = один **игровой месяц** фиксированной длительности в секундах.
- В режиме **play** клиент ведёт обратный отсчёт; по нулю — **переход к следующему периоду** (как при ручном `next`).
- **Пауза** останавливает течение периода в UI; синхронизация с сервером — `backend/app/game_time.py`.

### 3.2. Действия внутри периода

Примеры (не исчерпывающе):

- **Зарплата** — только по явному действию в текущем периоде; пропуск до конца периода → **выплата за период не повторяется** (сгорела).
- **Подушка** — переводы между `cash` и `safety_fund` с проверкой остатков (`balance_utils.adjust_safety_fund_balance`).
- **Активы** — покупка из шаблона БД (списание стоимости), удаление как продажа (зачисление стоимости).
- **Долги** — из шаблона: зачисление тела на счёт; закрытие: списание тела + просрочки; платёж за период — в конце периода (см. ниже).
- **События** — выбор варианта; эффекты из JSON; при нехватке средств — HTTP 400 с понятным `detail`.

### 3.3. Конец периода (`process_period_end`)

Реализация: `backend/app/game_period.py`. Последовательность по смыслу:

1. Списание **обслуживания активов** с `cash` (допускается отрицательный баланс как «неизбежные траты» на MVP).
2. Начисление **дохода от активов** (`monthly_income`) на `cash`.
3. **Платежи по обязательствам:** сумма к оплате = текущий месячный платёж + накопленная **просрочка**; с `cash` списывается сколько возможно; остаток уходит в `overdue_amount`.
4. **Страховки** — премии за период.
5. **Инвестиции** — капитализация депозита, купоны облигаций на `cash`.
6. Если `cash_balance < 0` — рост `negative_periods_count`; три периода подряд — **поражение** (`is_active = 0`); иначе счётчик сбрасывается.
7. Запись **снимка** в `PeriodEconomyClosing` для аналитики.
8. Увеличение `period_index`, обновление якоря времени.
9. **События** нового периода: `ensure_period_events` — до **трёх** сценариев; фильтр **`EventDefinition.mode`** (`game` / `plan` / `any`) + **`profile.save_kind`**; tier от **`period_index`** (см. §7.2, [`remove-character-xp-and-levels.md`](../vision/ideas/remove-character-xp-and-levels.md)); без дозаполнения после ответов в том же периоде.

**Итог цикла:** жить период → принимать решения → на границе месяца экономика **автоматически** применяет расходы/доходы/просрочки → новый период с новыми (или оставшимися) событиями.

## 4. Модель денег и обязательств

### 4.1. Два кошелька

- **Cash** — операционный счёт (платежи, покупки, события).
- **Подушка** — отдельный баланс; **условие победы MVP** завязано на соотношении подушки и ежемесячных обязательств.

### 4.2. Долги

- Поля: тело, годовая ставка %, рассчитанный **ежемесячный платёж** (в текущей формуле: доля годовых процентов на один период, см. `finance_helpers.monthly_interest_payment`), просрочка.
- Оформление из шаблона: **тело зачисляется на cash** (кредит «получен»).
- Закрытие: **возврат тела + просрочки** с проверкой ликвидности.

### 4.3. Активы

- Покупка: списание **стоимости** с `cash`.
- Обслуживание: списание в конце периода.
- Доходный актив: **доход в конце периода** без отдельной кнопки.
- Продажа (удаление): зачисление **стоимости** на `cash`.

---

## 5. События (MVP)

- Каталог: `EventDefinition` + `EventChoice` (`effects_json`: например `cash_delta`, `safety_delta`).
- На период — **ограниченное число** экземпляров `EventInstance`; после ответа карточка исчезает из pending, **новые в том же периоде не подмешиваются**.
- В схеме БД заложены расширения: `mandatory`, `category`, `metadata_json` (логика обязательных событий — на будущее).
- Назначение: **короткие сюжетные решения** с **немедленным** денежным следствием и проверкой средств.

---

## 6. Инвестиции и страховки (MVP)

- **Инвестиции:** депозит (капитализация по периодам), облигации (купон на баланс); позиции в БД, закрытие позиций.
- **Страховки:** ежемесячная премия в конце периода; виды и лимиты — задел под связку с событиями.

## 7. Победа, метрики, аналитика

### 7.1. Победа (MVP)

Считается в `GET /api/finance/overview`:

- `safety_fund_balance >= 3 * total_monthly_obligations` (обязательства = платежи по долгам + обслуживание активов);
- `total_overdue_amount == 0`;
- `net_monthly_cashflow >= 0`;
- **`win_reached` не бывает `true` при `period_index < 7`** — в первые шесть игровых периодов победа технически и по продукту недоступна (см. `MIN_PERIOD_INDEX_FOR_WIN` в `backend/app/routers/finance.py`).

Дополнительно в профиле ведётся **`clean_period_streak`** — серия периодов без просрочек (мотивация и аналитика).

**Целевая модель (Game):** несколько целей из шаблона, «M из N», средний чистый cashflow за последние 6 периодов — см. [evolution §II](../vision/ideas/money-quest-evolution-after-mvp.md#часть-ii--полная-концепция-после-обсуждения-qa-и-план-работ).

### 7.2. Прогрессия и мотивация (без character XP)

**С 2026-05-24:** в `GET /api/finance/overview` **нет** `score`, `gamification_level`, `xp_to_next_level`, `character_*`. Мотивация — **цели победы** из шаблона, **достижения** (без XP за tier), **период** и **чистая серия** (`clean_period_streak`). Сложность событий — **`event_tier`** по **`period_index`** (10 периодов = 1 band), см. [`remove-character-xp-and-levels.md`](../vision/ideas/remove-character-xp-and-levels.md).

### 7.3. Аналитика

- API: `GET /api/finance/analytics/timeseries` — ряд закрытий + текущая проекция.
- UI (премиум): цели, графики, нагрузка, наивный прогноз — **визуализация траектории**, не инвестиционная рекомендация.

## 8. UX-принципы

- Нижняя навигация по зонам игры.
- Премиум-оболочка (hero, карточки) — игровое ощущение, а не «учёт в Excel».
- События — оверлей с фокусом на решении; при смене периода возможен автопоказ pending-сценариев.
- Ошибки API — тосты (`notifications`), не блокирующие `alert`.

## 9. Техническая архитектура (кратко)

| Слой | Стек / подход |
|------|----------------|
| Backend | FastAPI, SQLAlchemy, PostgreSQL, JWT |
| Миграции | SQL-файлы + лёгкая авто-подстройка схемы в `main.py` (без Alembic) |
| Фронтенд | React, Vite, `@telegram-apps/telegram-ui`, хук `useGame` |

| Аудит денег | Таблица `transactions` |

Каталоги **шаблонов активов и долгов** — в БД (`asset_templates`, `liability_templates`), флаг `is_active`, выдача только активных записей.

## 10. Намеренные ограничения MVP и направления роста

- MVP: без налогов, сложной амортизации, штрафов за просрочку как отдельной каскадной системы; списания в конце периода **автоматические** (ручная оплата по статьям — задел в концепции, см. evolution §II).
- События: ограниченный набор эффектов; обязательные сценарии — поля в схеме, логика поэтапно.
- **Game / Plan**, шаблоны, движок победы из шаблона, расходы «жизни» + события, достижения — **приоритетный трек** (см. evolution §II.3).
- Связка событий со **страховками и типами активов**, штрафы/налоги — следующие слои после стабилизации контрактов.

## 11. Резюме

**ТВОЙ ХОД** — **симулятор месячного личного бюджета** в форме **игры в Telegram**: время задаёт ритм, деньги конечны, активы и долги меняют cashflow, события добавляют неопределённость, подушка и дисциплина ведут к **измеримой цели победы**, а аналитика помогает увидеть **динамику**, а не только снимок «здесь и сейчас». В целевой версии продукт явно разделяет **игровой** и **планировочный** режимы сохранения (см. §0). Направления углубления модели без смены этой рамки — в **§12**.

## 12. Улучшения: логика, сущности, механики, математика

Раздел фиксирует **качество и согласованность** текущей модели: что усилит реализм, обучаемость и поддерживаемость кода. Часть пересекается с [`PRODUCT_BACKLOG.md`](../backlog/PRODUCT_BACKLOG.md) и [evolution §II](../vision/ideas/money-quest-evolution-after-mvp.md); здесь акцент на **логике и цифрах**.

### 12.1. Логика и согласованность правил

| Тема | Суть проблемы | Направление |
|------|-----------------|-------------|
| Платёж по долгу | Сейчас по сути **проценты на тело**; тело не убывает от периодических платежей | Либо **явно** в продукте/UX назвать контракт «обслуживание долга / %», либо ввести **амортизацию тела** (аннуитет / дифференцированный график) |
| Условие победы | `3 × obligations` не отражает **нерегулярные** траты (события, крупные покупки) | Цель из **шаблона** (см. §0), скользящий burn, или метрика «подушка в месяцах расходов» по истории |
| «Сгоревшая» зарплата | Сильный педагогический приём, но бинарно | Опция **late claim со штрафом**, режим easy/hard, или событие «задержка выплаты» |
| Просрочка | Накопление без «давления» кроме суммы | Мягкие последствия: рост эффективной ставки после N периодов, блок новых кредитов, влияние на score — без обязательного мгновенного game over |

### 12.2. Сущности и данные

| Тема | Направление |
|------|-------------|
| Единый аудит денег | Все движения `cash` / `safety_fund` через **один слой** с типом и ссылками (`liability_id`, `event_instance_id`, …) — меньше расхождений с `Transaction` |
| Версии шаблонов | У `asset_templates` / `liability_templates` — **`valid_from` / `valid_to`** или версия, чтобы смена цифр в каталоге не ломала старые сохранения |
| Инвестиции и cash | Явно моделировать **перевод в principal** и обратно при закрытии как парные ноги — выписка «сходится» |
| Страховки | Связь **полис ↔ тип события / ущерб** и флаг «событие покрыто» — иначе страховка остаётся отдельным списанием |

### 12.3. Механики

| Тема | Направление |
|------|-------------|
| Обязательные события | Поле `mandatory` уже в схеме — правила: **блок закрытия периода**, автосценарий, или штраф за отказ |
| Лимиты на долг/активы | **DTI**, лига, уровень — кап на новые обязательства, анти-эксплойт шаблонов |
| Темп периода | Конфиг «игровой минуты» vs «календарный день» без переписывания ядра экономики |
| Условия поражения | Не только отрицательный `cash`: порог **просрочки / DTI**, или **restructuring** вместо одного game over |

### 12.4. Математика и расчёты

| Тема | Направление |
|------|-------------|
| Float | Перевод денежных полей в **`Decimal`** или **целые копейки** + единая банковская округля на границе API |
| Формула долга | Один модуль (`finance_helpers` + тесты), без дублирования логики в SQL/миграциях |
| Epsilon | Единые пороги «ноль / хватает» для сравнений и UI |
| Ограничения механик | Через **шаблоны старта** (blueprint / flags), не через character level; tier событий — от **`period_index`** |
| Прогноз в аналитике | Линейная экстраполяция — только с **дисклеймером** или сценариями («без событий», «стабильный поток») |

### 12.5. Аналитика и обратная связь игроку

- **Разложение** `net_monthly_cashflow`: вклады зарплаты, активов, долгов, обслуживания, страховок, среднего по событиям — чтобы видеть **рычаги**, а не одно число.
- Явное объяснение **`clean_period_streak`**: что считается «чистым» периодом после частичной просрочки.

### 12.6. Приоритизация (ориентир)

| Приоритет | Направление |
|-----------|-------------|
| Высокий | Деньги: целые копейки / `Decimal`, единые округления |
| Высокий | Единая модель **прогресса и очков** в UI и API |
| Средний | Долги: реалистичное убывание тела **или** честный «процент-only» контракт в копирайте |
| Средний | Обязательные события; связка **страховка ↔ событие** |
| Ниже | DTI-лимиты, расширенные поражения/реструктуризация, сценарный прогноз |

## 13. Целевой игрок, длительность заходов и контентные рамки (MVP 1.1+)

Каноничные решения продукта (ЦА 14–35, новички в финграмотности через игры, типичная сессия ~1–3 мин сейчас и рост до 10–15 мин, избегаемые темы, мотивация «победа/уровень», паттерны текста событий) собраны в **[`foundation/TARGET_PLAYER_AND_SESSION.md`](TARGET_PLAYER_AND_SESSION.md)** и должны проверяться при спецификации событий, прокачки и нового контента.
